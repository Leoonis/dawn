### ADR-18: Flex-child width pattern for Shopify custom-element wrappers (visual color bugs can be sizing bugs)

| Field | Content |
|---|---|
| ID | ADR-18 |
| Date | 2026-05-22 (Task 064a hotfix journey, commit cd064e4) |
| Decision | When a Shopify custom element (Web Component like `<sticky-header>`) is placed inside a flex parent with `display:flex` direction:column, the custom element defaults to `flex-grow:0` and does NOT extend to parent width. Wrapper class (e.g., `.header-wrapper`) must be given explicit `flex: 1; width: 100%;` to occupy full parent width. This is the canonical pattern for ALL Dawn custom-element wrappers inside flex layouts (sticky-header, sticky-cart, sticky-banner, predictive-search-modal, etc.). |
| Rationale | Task 064a hotfix journey isolated the real root cause through 3 attempts over 3 commits: (1) commit 343ae10 attempted background-color fix with invalid `var(--color-background)` syntax — INTRODUCED REGRESSION making header transparent (invalid var() resolves to CSS initial value, NOT prior cascade; mechanism in Issue 16); (2) commit a6b53b8 wrapped token in `rgb()` for valid CSS — restored opacity but the original "shelf at right edge" symptom persisted; (3) commit cd064e4 isolated via DevTools console scripts (`document.querySelector(".header-wrapper").getBoundingClientRect()`) that wrapper width was 1208px in a 1280px viewport — the bug was layout SIZING, not color. # Voice Canon — Noctorya

The complete reference for how Noctorya sounds. Cowork should treat this as binding for all Noctorya writing tasks (product descriptions, "Story Behind" essays, artist bios, blog posts, future content additions). If a piece of copy doesn't pass the tests in this document, it's not Noctorya.

---

## Why this voice and not another

Noctorya's voice is positioned as **"cold authority"** — the register of a contemporary critic whose judgment is not negotiable. The voice assumes the buyer is intelligent and refuses to coddle. The brand temperature is cool throughout: detached, fact-loaded, uninterested in performing warmth.

**Axis 1: Authority register.** Noctorya sits at ~80% toward academic — credentialed-feeling, scholar-distance, no warm curatorial affection. The writing knows the field but doesn't perform expertise; it simply states what is.

**Axis 2: Emotional temperature.** Noctorya sits at ~30% toward emotional. The writing acknowledges that paintings exist and have effects, but it does not invite the buyer to feel anything. The reader does the emotional work alone. Noctorya describes the picture; the response belongs to the viewer.

**Axis 3: Sentence music.** Noctorya sits at ~40% toward writerly. Sentences are short and declarative. Periods do heavy lifting. The rhythm is staccato, not flowing — the cadence of confident judgment, not lyrical observation.

The combined position: **a contemporary critic who has decided, and is telling you what they decided**. This is the register of: serious art-world publications (Frieze, Artforum reviews), Manhattan gallery press releases for blue-chip exhibitions, MoMA wall labels (NOT Met wall labels — MoMA uses contemporary-minimal language). It is NOT the register of: Saatchi marketing copy (too sales), Pinterest captions (too friendly), pop-art-history podcasts (too narrative), Tumblr essays (too emotional).

Noctorya sounds like a person who has stopped explaining why the work matters, because anyone who needs that explanation isn't the buyer.

---

## Why not the alternatives

Three alternative voices were considered and rejected.

**Pure minimalist-cold (one-line descriptions).** "Repin. Ivan the Terrible. 1883. Tretyakov Gallery." This is the most extreme Noctorya could go — pure museum label, no narrative. Rejected because it fails SEO (Google needs body content for indexing) and because it tells the buyer nothing they couldn't get from a postcard. Noctorya must still be a STORE, which means there must be enough content to convert. The minimalist register is right; the minimalist length is too thin.

**Aspirational-luxury (fashion brand copy).** "An object of profound stillness. A meditation on the limits of beholding. Noctorya invites you to observe." Rejected because it parodies itself. Luxury copy is mocked online for being content-free. Noctorya's authority must be earned by FACTS — dates, museums, technique notes — not by adjective-stacking. The luxury register signals "we have nothing real to say" and Eleanor-persona buyers detect this instantly.

**Warm-curatorial (scholar with affection).** "In the hush of Schenck's pastoral, the ewe keeps a vigil older than language..." Beautiful for one essay. Wrong for Noctorya. Warm voice signals "I want you to feel what I felt"; Noctorya signals "the painting is here, the facts are here, decide for yourself." Warm voice with literary motifs would also be a thesaurus pass at scale — exhausting at 20 paintings. Noctorya's identity comes from austerity, not from flourish.

The chosen Noctorya voice keeps the academic-fact spine but writes it in short sentences, with cool vocabulary, and with NO literary motifs. The signature is the ABSENCE of signature.

---

## What feeling Noctorya's buyer should have reading a product page

Call her **Eleanor**. Eleanor is 33, design director at a creative agency, lives in a converted loft in a second-tier city (Lisbon, Berlin, Mexico City, Toronto). She earns mid-six figures, spends less time online than her younger peers, and reads physical books. She has been to MoMA twice this year. She owns one piece of art over $1,000. She is not impressed by pretty writing. She is impressed by the absence of fluff.

Beat by beat, Eleanor reading a Noctorya product page:

1. **First 5 seconds (image + title):** "Striking. I want to see it framed."
2. **Reading the artist line:** "I know this artist. Or I should."
3. **First sentence:** "Direct. Good. They're not selling."
4. **Mid-essay:** "These are facts I can verify. I trust them."
5. **Final sentence:** "OK. They're done. The painting speaks the rest."
6. **Variants + price:** "Reasonable for what this is. They're not gouging."
7. **Add to cart:** "I'll buy it because I want it. Not because they convinced me."

Noctorya's success state: the buyer feels she made the decision herself. The copy was a fact-check, not a persuasion. Noctorya must NEVER produce the feeling that the writer is trying to make her buy. Trying ruins the brand.

---

## What's "off-brand" if anyone writes Noctorya themselves

Noctorya has specific failure modes. If anyone (founder, contractor, AI without canon access) writes Noctorya content, watch for these:

- **Any literary flourish.** Phrases like "in the hush of...", "what stays with you...", "older than..." are FORBIDDEN. Noctorya has no motifs. Its signature is the absence of signature.
- **Second-person address.** "You stand before..." is a warm-voice move. Noctorya stays third-person throughout. The buyer is not addressed.
- **Emotional verbs about the buyer.** "You will feel..." or "moves the viewer to..." — Noctorya does not predict the response.
- **Adjective stacking.** "A profound, haunting, deeply moving meditation..." — three adjectives is two too many. Noctorya uses one when it must, zero when possible.
- **Explanation of importance.** "This painting is important because..." — if it's important, the facts demonstrate it. Don't tell.
- **Coyness, irony, or wink.** Noctorya is sincere and direct. No "iconic" used ironically, no "obviously" with hidden smile. The voice is straight.
- **Sentimentality about the artist.** "Tragic figure", "tortured genius", "misunderstood master" — Noctorya reports facts about lives, not narratives about suffering.

If Cowork sees these in drafts, push back with the canon reference.

---

## The five-line voice test

Any new Noctorya copy passes if and only if all five of these are true:

1. **Could this paragraph appear unchanged in a MoMA wall label or Frieze review?** If yes, it's Noctorya. If it would be too soft for those contexts, no.
2. **Is every sentence factually verifiable?** Noctorya tolerates almost zero interpretation. The interpretive lift is the buyer's job.
3. **Are there ZERO literary flourishes?** Noctorya has no motifs, no rhythmic flourishes, no closing turns. It ends when the facts end.
4. **Is the average sentence under 20 words?** Noctorya is short-sentence-default. Average ~14-16 words per sentence. Periods do work.
5. **Is the tone consistently cool?** No warm verbs, no curatorial affection, no "we love this" energy. Cool throughout.

## Banned vocabulary

The Noctorya banned vocabulary is organized in two parallel layers. Both apply to all customer-facing copy. Layer A targets voice-register failures (literary register where Noctorya should be flat). Layer B targets AI-slop failures (LLM-marketese where Noctorya should be human-edited prose). Any draft that hits a word from either layer fails voice check and requires revision.

### Layer A - Voice-canon banned (literary register)

- Empty intensifiers and cliches: haunting, captivating, breathtaking, stunning, must-have, iconic, mesmerizing, unforgettable
- Empty intensifiers (modifiers): truly, simply, absolutely
- Visual decoration: emoji, ALL CAPS for emphasis, exclamation marks except in quotation
- Literary motifs: "in the hush of...", "the gravity reserved for...", "older than...", "what stays with you...", "current" used as adjectival closer. These are warm-voice brand markers. Noctorya must not borrow them.
- Second-person pronouns: "You", "your" - banned in body copy. Only acceptable in headings or CTAs.
- Emotional adjectives applied to the work: "Moving", "powerful", "stirring", "profound", "deeply", "intense". Noctorya describes facts. Adjectives of effect are off-brand.
- Critical-judgment words: "Important", "significant", "major", "essential". Words that judge the work's place in art history. Noctorya reports the critical reception, doesn't make it.
- "Master" / "masterful" / "mastery": warm credentialing words. Noctorya avoids them. "Skilled" or factual technique notes are fine.
- Aesthetic compliments: "Beautiful", "gorgeous", "magnificent". Noctorya doesn't compliment the work; that's the buyer's call.
- Atmospheric cliches: "Enchanting", "magical", "dreamlike". Noctorya stays concrete.
- Explanatory phrases: "What this means is...", "In other words...", "Put simply...". Noctorya doesn't translate. The reader keeps up or doesn't.

### Layer B - AI-slop banned (cross-format)

These markers are not literary-register failures but LLM-marketese fingerprints. They signal AI-generated text that has not been human-edited into Noctorya voice. They apply to product descriptions, blog posts, social captions, email copy, theme microcopy, internal-facing docs that may later be repurposed for customer-facing use.

- intimate
- captures the silence of
- delve / delving
- boundaries
- tapestry / rich tapestry
- in today's fast-paced world
- game-changing
- leverage / leveraging
- unleash
- supercharge
- masterpiece
- elevate (verb)
- embark on a journey

When in doubt: would this word appear in a 200-word Frieze review of a contemporary exhibition? If yes, fine. If it would feel out of place there, suspect it.

---

## Signature traits (not motifs — Noctorya has no motifs)

Noctorya's "signature" is structural rather than phrasal. The recognizable Noctorya moves:

- **The cataloging opener.** First line: title, artist, medium, year. Period-separated. Like a museum card. This is Noctorya's most consistent move and the strongest first-read brand identifier.
- **Sentences ending on facts, not feelings.** The last sentence of a paragraph names a date, a museum, a number, a verifiable event. Never names a feeling.
- **Unsentimental endings.** Essays end on small concrete facts that carry weight by implication, not by writerly sentence-music. "The image itself has not changed." "Visitors photograph it more often than any other painting." "It has been reproduced on the interior of the Tampere highway tunnel."
- **Refusal of synthesis.** Noctorya does not deliver a thematic conclusion. The facts are presented, the essay stops. The reader synthesizes.
- **Short paragraphs.** Two to four sentences each. White space between paragraphs is part of the brand.

These structural moves repeat across essays without becoming repetitive, because they are FORMS rather than PHRASES. Noctorya repeats forms (structures), not phrases (motifs).

## Title and URL conventions

Title format: `[Artist Last Name]: [Painting Title]`
URL slug: `artist-lastname-painting-name` (e.g. `rembrandt-night-watch`)

These are SEO-relevant choices. Title and slug structure must be consistent across the catalog so Google reads the store as a coherent inventory of Old Masters by named artists.

## What Cowork should do when in doubt

If a draft is ambiguous on voice — feels in Noctorya range but uncertain — do this:

1. Read the five-line voice test. Pay special attention to point 4 (avg sentence under 20 words) and point 3 (zero flourishes).
2. Compare the draft to the closest reference essay above. Ask: is it as cold? As short-sentenced? As fact-loaded?
3. Read it aloud. If it sounds like persuasion, it's drifted. If it sounds like a museum-label-with-context, it's right.
4. If still unsure, raise to the founder.

The cost of voice drift in Noctorya is high. The catalog is small (20 paintings at launch); every essay is ~5% of the brand voice. Noctorya's brand specifically depends on austerity holding — one warm essay in a cold catalog breaks the spell more visibly than the inverse, because cold voices are harder to maintain and easier to violate by accident.

### capture vs assign

Use `{% capture %}` for multi-step string composition; use `{% assign %}` for
single-value bindings or single-filter operations.

```liquid

{%- assign primary_color = settings.color_accent -%}
{%- assign sorted = collection.products | sort: 'price' -%}
```

Avoid `{% assign %}` for multi-step concatenation (works but is harder to read than capture).

### render not include

Liquid `{% include %}` is deprecated. Always use `{% render %}` for snippets.

CORRECT: `{% render 'card-product', product: product, lazy: true %}`
INCORRECT: `{% include 'card-product' %}` (deprecated; may fail in future Shopify versions)

`{% render %}` accepts named parameters and isolates scope. Pass everything the snippet
needs explicitly.

{%- for block in section.blocks -%}

## CSS ARCHITECTURE

### BEM convention

CSS class naming follows BEM (Block - Element - Modifier). Block names use kebab-case;
elements attach to the block with double underscore (`__`); modifiers attach to either
a block or an element with double hyphen (`--`). Names describe ROLE, not appearance.

**Correct patterns**

```css
.product-card                           /* Block - standalone component */
.product-card__title                    /* Element - belongs to .product-card */
.product-card__price                    /* Element - sibling element on same block */
.button--primary                        /* Modifier on a block - variant of .button */
.product-card__title--featured          /* Modifier on an element - state of a title */
.site-header__nav                       /* Hyphenated block + element - both valid */
.cart-drawer__line-item                 /* Multi-word element - kebab-case inside __ */
.button--primary.button--large          /* Composition - two modifiers on one block */
```

Per-example pedagogical notes:

- `.product-card` - Block is a self-contained component. Two-word name in kebab-case;
  no underscores yet because no element or modifier is involved. A block can stand
  alone in HTML without any siblings.
- `.product-card__title` - Double underscore signals "this element BELONGS to the
  block". The block name is repeated in full (never abbreviated); the element name
  is kebab-case after `__` (write `__line-item`, not `__lineItem` and not `__li`).
- `.product-card__price` - Sibling element. Multiple elements share the same block
  prefix; each describes a distinct part. The element name describes function, not
  position (write `__price`, not `__bottom-text`).
- `.button--primary` - Double hyphen signals "this is a VARIANT of the block". The
  block name is preserved in full; the modifier describes the variant by purpose
  (`primary`, `secondary`, `ghost`), not appearance (`red`, `bold`, `big`).
- `.product-card__title--featured` - Modifier attached to an ELEMENT, not the block.
  Triple-segment (`block__element--modifier`) is the legitimate maximum nesting depth
  in BEM. Anything deeper is a sign you need a new block.
- `.site-header__nav` - Hyphens are allowed INSIDE a block name (`site-header`); the
  element separator is still double underscore. Do not confuse `-` (word separator)
  with `__` (block-element separator).
- `.cart-drawer__line-item` - Multi-word elements also use kebab-case after the `__`.
  The `__` only separates block from element; the element name itself stays
  kebab-case for any internal word breaks.
- `.button--primary.button--large` - Composition via two modifiers stacked on the
  same element in HTML. Each modifier is applied as a SEPARATE class; you NEVER
  write `.button--primary--large` (which would imply "primary-large" is a single
  modifier name). In HTML:
  `<button class="button button--primary button--large">SHOP</button>`.

**Incorrect patterns**

```css
.productCardTitle                       /* camelCase - violates kebab-case rule */
.product_card_title                     /* Single underscores - ambiguous, not BEM */
.product-card-title                     /* Single hyphen between block and element */
.product-card__title__featured          /* Double element nesting - forbidden */
.product-card--title                    /* Modifier syntax used for an element */
.red-title                              /* Color in name - not semantic */
.title-featured                         /* Modifier without parent block - dangling */
.button--primary--large                 /* Two modifiers fused - should be separate */
```

Per-example pedagogical notes:

- `.productCardTitle` - camelCase violates BEM's kebab-case rule. Switch to
  `.product-card__title`. CamelCase is a JavaScript identifier convention, not a CSS
  class convention in this repo.
- `.product_card_title` - Single underscores are ambiguous between word-separator
  and element-separator. BEM uses kebab-case (`-`) for words and double underscore
  (`__`) for the block-element boundary. Pick one role per delimiter and stick to it.
- `.product-card-title` - Single hyphen between `card` and `title` could mean either
  "the title of a card" or "a card-title component in its own right". BEM removes
  that ambiguity with `__`. Switch to `.product-card__title`.
- `.product-card__title__featured` - Elements DO NOT nest in BEM. If you find
  yourself writing `__x__y`, the deeper segment is either a modifier
  (`--featured`) or a separate block (`.featured-badge`) that happens to live
  inside the card.
- `.product-card--title` - Modifiers are for VARIANTS, not for elements. The `--`
  syntax describes "a variant of the parent thing"; an element is a different
  concept and uses `__`. Switch to `.product-card__title`.
- `.red-title` - Names should describe ROLE, not APPEARANCE. If the design rebrands
  and red becomes silver, the class name is now wrong. Use
  `.product-card__title--featured` (purpose) or `.product-card__title--accent`
  (semantic emphasis) so the class survives a visual refresh.
- `.title-featured` - Modifiers are valid only when attached to a block or element
  they modify. A free-floating modifier has no parent and cannot exist alone.
  Either attach it (`.product-card__title--featured`) or promote the concept to a
  block in its own right (`.featured-callout`).
- `.button--primary--large` - Two modifiers cannot be fused with `--`. Each modifier
  is independent; in HTML, apply them as two separate classes:
  `<button class="button button--primary button--large">`. In CSS, target each
  modifier with its own selector. Fusing them creates exponential class growth as
  variants multiply.

**Edge case: utility classes are NOT BEM**

Single-purpose utility classes (`.u-hidden`, `.text-center`, `.mt-md`, `.is-active`,
`.has-error`) live in a separate namespace and DO NOT follow BEM. They are
intentionally short, prefixed (`u-`, `is-`, `has-`) or descriptive
(`text-center`), and applied directly in HTML alongside BEM block classes:

```html
<article class="product-card is-loading">...</article>
<p class="product-card__price text-center">...</p>
```

Do NOT retrofit BEM onto utility names (`.product-card__u-hidden` is wrong on both
counts). Do NOT mix BEM and utility conventions in the same class name. Keep the
two namespaces separate by convention; this repo's utility namespace is documented
in `assets/base.css` and `assets/brand-tokens.css` comments.

### Mobile-first @media pattern

Default styles target mobile (0-749px); tablet and desktop layered via min-width:

```css

@media (min-width: 750px) { .product-card__title { font-size: var(--fs-body-tablet); } }

Example: `sections/trust-bar.liquid` -> `assets/trust-bar.css`.

```

NEVER override Dawn variables outside brand-tokens.css. NEVER hard-code Dawn's

## ACCESSIBILITY

### Semantic HTML

Use semantic elements first; add ARIA only when semantics are insufficient:

- `<header>`, `<nav>`, `<main>`, `<section>`, `<article>`, `<aside>`, `<footer>`
- `<button>` for interactive triggers (NEVER `<div onclick>`)

### WCAG 2.4.7 focus indicators

ALL interactive elements must have visible focus. Both `outline` AND `box-shadow`
required (Windows High Contrast Mode suppresses shadows):

```css
:focus-visible {
  outline: var(--focus-outline-width) solid var(--focus-outline-color);
  outline-offset: var(--focus-outline-offset);
  box-shadow: 0 0 0 var(--focus-ring-width) var(--focus-ring-color);
}
```

NEVER use `outline: none` without immediate replacement. Tokens encode WCAG-compliant values.

### Keyboard navigation

- All interactive elements reachable via Tab.
- Tab order matches visual order (do NOT use `tabindex` to reorder).
- Esc closes modals/drawers.
- Enter and Space activate buttons.
- Arrow keys navigate within a custom component (carousel, accordion) when component owns focus.

### Touch targets

Minimum 48x48 px on mobile interactive elements (link, button, icon button).
Below 48px, add invisible padding via `:before` or padding utility.

### Color contrast

WCAG AA validated combinations documented in `assets/brand-tokens.css` comments and
in `design_system.md` Section 1 WCAG Contrast Verification table. Do not pair tokens
outside the validated set.

### Alt text and labels

- Image alt text: 80-125 chars, factual, includes artist + title + product type.
- Decorative images: `alt=""` (empty, not omitted).
- Form labels: visible `<label>` preferred; `aria-label` only when visible label is impossible.

## 9. DAWN ARCHITECTURE LEARNINGS (Tasks 045-053 history)

Detailed mechanism documentation surfaced during Phase D Foundation Layer build. All mechanisms verified by direct file inspection.

### 9.1 Body class hardcoded mechanism

**Mechanism:** `dawn/layout/theme.liquid` line 302 hardcodes the body class. Body element NEVER receives `color-scheme-N` class automatically. Section-level color schemes apply via the section wrapper div, not the body.

**Implication:** Body background color comes from the `--gradient-background` variable (Section 9.3) which Dawn emits in an inline `{% style %}` block from `settings_data.json` scheme settings. Body cannot inherit a color scheme by class.

**Section render pattern (Dawn convention):**

```liquid
<div class="... color-{{ section.settings.color_scheme }} gradient">
```

The wrapper div carries both the `color-scheme-N` class AND the `gradient` class, both required for the dark theme to render correctly.

### 9.2 .gradient class cascade

**Source:** `dawn/assets/base.css` lines 2934-2938.

`.gradient` class uses `var(--gradient-background)` NOT `var(--color-background)`. This is a deliberate Dawn architecture choice to allow gradient overrides without affecting solid color rendering.

### 9.3 --gradient-background variable emission point

**Source:** `dawn/layout/theme.liquid` lines 64-68.

Dawn emits `--gradient-background` from `scheme.settings.background` admin value via inline `{% style %}` block at the top of `<head>`. This means the admin-side scheme color drives the inline CSS that sets body background.

**Implication for Noctorya:** To force the body to render Noctorya black (#111111), BOTH must be aligned:
1. `dawn/assets/brand-tokens.css` defines `--color-background: 17, 17, 17` in the override block
2. `dawn/config/settings_data.json` scheme-1 admin colors must be set to Noctorya values so Dawn's inline style emission also produces #111111

Task 051 Phase 5 aligned both. Single Dark Uniform = scheme-1 forced everywhere.

### 9.5 Shopify CLI non-ASCII restriction

**Rule:** Shopify CLI rejects ANY non-ASCII byte in `.liquid` files. This includes UTF-8 multibyte sequences like:
- middle-dot `·` (U+00B7, encoded as bytes `C2 B7`)
- em-dash `—` (U+2014, encoded as bytes `E2 80 94`)
- ellipsis `…` (U+2026, encoded as bytes `E2 80 A6`)

**Workaround:** Use HTML entities instead. The CLI does not parse entities; it only scans raw bytes.
- `·` -> `&middot;` or `&#183;`
- `—` -> `&mdash;` or `&#8212;`
- `…` -> `&hellip;` or `&#8230;`

**Scope of restriction:** Applies ONLY to `.liquid` files. Does NOT apply to:
- `.css` files (free to use any UTF-8 byte)
- `.json` files (Shopify JSON parser handles UTF-8 natively)
- `.svg` files (XML-encoded entities are interpreted)

**Discovery commit:** Task 053-fix (SHA 2aa3d9d). Middle-dot literal `·` in `sections/trust-bar.liquid` line N caused Shopify CLI sync rejection; replaced with `&middot;` HTML entity.

### 9.6 Line endings

**Dawn baseline:** CRLF universally across all original Dawn files (Shopify upstream uses Windows-style line endings for Liquid theme tradition).

**Noctorya-authored files (via Python Write):** LF-only. Tools used:
- Python `pathlib.Path.write_text()` produces LF
- Cowork bash heredoc `cat > file` produces LF
- Shopify CLI TOLERATES LF-only files in sync

**Git handling:** `core.autocrlf=true` configured at repo level. Commits land with CRLF in git index, LF in working tree on Windows-side checkouts. Cowork sandbox (Linux bindfs mount) writes LF and git stages it without auto-conversion.

**313 modified-file warning:** `git status` shows 313 files modified after Cowork operations even when no semantic changes were made. This is the CRLF/LF normalization drift, NOT actual file content changes. Benign per discovery during Phase D Task 045 setup.

### 9.7 JSON config edit drift

**Discovery:** Python `json.dumps()` round-trip causes byte-level changes on unrelated lines in `settings_data.json` and `templates/index.json`. Example: `</p>` becomes `<\/p>` (forward-slash escape removal/addition) on lines NOT touched by the edit.

**Reason:** Different JSON serializers normalize escape sequences differently. Dawn baseline used specific escape conventions; Python's default `json.dumps` uses different ones.

**Impact:** JSON-equivalent (Shopify parses both forms identically); functionally harmless. BUT breaks strict byte-identity which can confuse diff-based tools.

**Workaround for byte-strict tasks:** Use byte-level regex insertion via `re.sub()` on the raw text WITHOUT JSON parse-and-reserialize cycle. Apply only when strict byte preservation is required.

### 9.8 Foundation Layer mechanism (Tasks 050-051 combined)

**Load order in `dawn/layout/theme.liquid`:**
1. `assets/base.css` (Dawn baseline)
2. `assets/brand-tokens.css` (Noctorya tokens at line 259 stylesheet_tag)
3. Section-scoped CSS via `{{ 'section-name.css' | asset_url | stylesheet_tag }}` per section

**Result:** Noctorya's `:root` redeclarations override Dawn's defaults by source order. Dawn's inline `{% style %}` block (from settings_data.json scheme-1 admin colors) emits the second override layer. Both must align for full Noctorya rendering.

### 9.9 Dawn baseline indentation conventions

- `theme.liquid`: 4-space CRLF
- JSON files (settings_data.json, index.json, etc): 2-space CRLF
- `base.css` and other CSS: 2-space LF
- Liquid section files: 2-space CRLF mostly

When editing Dawn baseline files, preserve the existing indentation style at the edit point. When creating new files (custom sections, snippets), use 2-space LF (Noctorya convention; aligns with most CSS/JSON tooling).

### 12.3 Python heredoc gotchas

**Single-quoted PYEOF (`<< 'PYEOF'`):** internal single quotes in Python strings will trip the parser. Use it ONLY when no single quotes appear in the embedded Python.

**Double-quoted PYEOF (`<< "PYEOF"`):** safer for embedded Python with single quotes, but variable expansion `$VAR` happens before Python sees the script. Use it when Python code contains single quotes but no shell variables.

**Fallback for complex multi-file ops:**
1. Write tool: create new files in one shot (bypasses heredoc).
2. `python -c "..."` with simple replacement for short edits.
3. For long edits, write the Python script to a temp file via Write tool, then `bash python3 /tmp/edit.py`.

### 12.4 Edit-tool ban on Windows-mount

**Rule:** ALWAYS use Python in-place modification or Write tool. NEVER Edit tool on Windows-mounted files.

**Reason:** Edit tool has truncation bug on Windows file system mounts (bindfs interaction). Observed pattern: byte-count parity (pre and post identical) despite content expansion. Content past byte ~N silently lopped off, file ends mid-line at arbitrary character. Recovery requires restoring tail from backup.

**Defensive depth pattern (when editing is necessary):**
1. sha256 pre + post
2. JSON parse pre + post (if JSON file)
3. assert count==1 before any replace (catch over-replacement)
4. write only after all assertions pass
5. tail -3 post-write (catch silent truncation)
6. Git commit immediately if successful (versioned recovery point)

### 12.5 CRLF handling

- `core.autocrlf=true` at git config level (Eduard's Windows machine)
- Commits land with CRLF in git index, LF in working tree on Windows checkouts
- Cowork sandbox (Linux mount) writes LF; git stages LF without auto-conversion
- Warnings "LF will be replaced by CRLF" are benign
- Shopify CLI tolerates BOTH for sync
- DO NOT try to "fix" the 313-modified-file CRLF drift in `git status`; it is the expected baseline state.

14. **CSS specificity battle: Dawn `:not()` compound selectors require class chaining to override.** Discovered during Task 064a fix-pass (commit `5930e5c`). Dawn base.css uses high-specificity compound selectors like `header.shopify-section:not(.shopify-section-group-header-group) > nav .header__icon:not(.header__icon--account)` to scope icon visibility. Single-class overrides (e.g., `.header__icon { display: block; }`) lose the specificity battle and have no effect. Workaround: chain matching classes in the override selector to equal or exceed Dawn's specificity, OR target the specific compound (e.g., `.header__icon--account` plus parent class chain). Verify via DevTools "Computed Styles" panel which rule actually wins. Affects any future task overriding Dawn icon visibility, nav-link states, or component-state CSS.

15. **Python in-place edits on Windows-mount must detect line endings before writing.** SESSION_STATE.md is LF-only (Cowork writes from Linux bindfs mount). Dawn baseline files (theme.liquid, *.liquid, *.json) are CRLF (Windows-side Shopify convention). Python `pathlib.Path.write_text()` and `open(mode="w")` use the OS default line separator (LF on Linux mount), which silently converts CRLF source files to LF on write-back, breaking byte-identity with origin-repo state. Auto-detect pattern: read as bytes via `read_bytes()`, count `b"\r\n"` vs `b"\n"`, assert mode (LF/CRLF/MIXED), preserve mode on write back via binary I/O (`Path.write_bytes()`). Cowork pattern for SESSION_STATE.md is LF-only (file is sandbox-authored); for any future task editing Dawn baseline files, MUST detect-and-preserve EOL or use shopify theme dev sync which handles EOL transparently.

16. **CSS `var(--token)` requires `rgb()` wrap when token is a bare RGB triplet (Dawn convention).** Dawn defines color tokens as comma-separated RGB triplets (e.g., `--color-background: 17, 17, 17;`) and references them via `rgba(var(--color-background), 1)` or `rgb(var(--color-background))`. Bare `var(--color-background)` in a CSS property expecting a color value resolves to the invalid string `"17, 17, 17"` which fails CSS parsing AND resolves to the property INITIAL VALUE (not the prior cascade), producing transparent backgrounds, default colors, etc. Symptoms can masquerade as cascade issues. Always wrap Dawn RGB triplets: `background-color: rgb(var(--color-background));` or `rgba(var(--color-background), 0.88);`. Discovered during Task 064a hotfix attempt 1 (commit `343ae10` - INTRODUCED REGRESSION); fixed in `a6b53b8`. Related Web Component gotcha discovered same commit chain (now ADR-18): Shopify custom elements like `<sticky-header>` default to `flex-grow:0` inside flex parents and do NOT extend to parent width; explicit `flex: 1; width: 100%;` on the wrapper class is required. **Lesson generalizes:** "visual color bugs" with edge symptoms (shelves, gaps, partial backgrounds, transparent fills) can be SIZING/LAYOUT bugs in disguise; **DevTools console scripts (`getBoundingClientRect()`, `getComputedStyle()`) are higher-signal than screenshots for layout diagnostics.**
